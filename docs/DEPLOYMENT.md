# Deployment

How Agent Atlas ships (plan §17). CI pipeline: `.github/workflows/ci.yml`; security
posture: plan §15 + the CSP notes below.

## The pipeline

Every push and pull request runs, in order:

1. `npm run typecheck` — TypeScript + Astro diagnostics
2. `npm run validate` — content integrity: schema → graph → template lint →
   deterministic `graph.json`
3. `npm test` — unit + component tests
4. `npm run build` — static site (CSP hashes generated per page)
5. `npm run test:e2e` — Playwright against the built site; because the CSP is a
   browser-enforced `<meta>` policy, this doubles as a live CSP regression test

A failure at any step stops the job, and the deploy job `needs` the CI job — **a red
validate blocks deploy by construction**. Content errors are build errors.

## Enabling deployment (one-time, account-side)

The deploy job is skipped until explicitly enabled:

1. Create a Cloudflare Pages project named `agent-atlas`
   (dashboard → Workers & Pages → Create → Pages, "Direct Upload"; or
   `wrangler pages project create agent-atlas`).
2. Create an API token with the **Cloudflare Pages — Edit** permission and note the
   account id.
3. In the GitHub repo settings:
   - secrets `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`;
   - repository **variable** `CLOUDFLARE_DEPLOY_ENABLED` = `true`.

From then on: pushes to `main` deploy production; every other branch deploys a preview
URL (`<branch>.agent-atlas.pages.dev`) — the review surface for content intake changes.

## Portability (standing condition, DECISIONS.md)

Nothing in the application is Cloudflare-specific:

- The site is plain static output (`dist/`); any static host serves it.
- The **CSP lives in the HTML** as a per-page `<meta>` element generated by Astro
  (`astro.config.mjs` → `security.csp`) with hashes for Astro's inline island loaders —
  it holds identically on any host.
- `public/_headers` carries only the directives browsers ignore in meta CSP
  (X-Frame-Options, nosniff, referrer/permissions policies). The `_headers` convention
  is shared by Cloudflare Pages and Netlify; on Vercel the same five headers translate
  to a `vercel.json` `headers` block; on plain nginx, five `add_header` lines.
- The only Cloudflare-coupled artifact is the ~10-line deploy job in `ci.yml`; swapping
  hosts means swapping that job.

## Security posture summary

- Strict CSP: `default-src 'self'`, hashed inline scripts/styles only, no third-party
  origins anywhere (there are none to allow — no analytics, no CDNs, no fonts).
- No secrets exist in the deployed site or CI beyond the host deploy token (plan §15).
  The future real-model experiment key stays in a local git-ignored `.env`
  (ADR-0005) and is never needed by CI.
- Adding ANY third-party script/origin requires editing the CSP deliberately — the
  default posture makes drift visible in review.
